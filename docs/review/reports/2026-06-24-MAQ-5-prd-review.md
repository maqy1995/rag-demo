## 阻塞项（必须改才能进入开发）

### B1. 定位漂移：把"问答"当唯一入口会错失真正的核心价值
PRD §1.2 把产品定位成"可对话的外脑"，§3.1 F1-F7 全部围绕"问答"展开。
Obsidian 重度用户的**第一痛点不是"问 → 答"，而是"找 → 定位"**——
他们打开这个工具的第一动作是"我记过一条讲 X 的笔记在哪"，不是"给我
讲讲 X"。当前 PRD 完全没有把"纯语义搜索 / 召回列表"作为一等公民。

- 影响：MVP 跑通后用户大概率觉得"没比 Obsidian 自带搜索强多少"，原型验证失败。
- 建议（v0.1 必须做）：把 F4 检索升级为 F4' **Search + Ask 双面板**：
  - 左栏"列表视图"：Top-K 命中按相关度排序，标题 + snippet + 文件路径，点开就是原文
  - 右栏"问答视图"：选中列表项后可选"基于这 N 条给我讲一下"
  - 这样 LLM 完全是"可选项"，核心价值（语义搜索 + 溯源）不依赖 LLM 质量

### B2. "30 分钟首次启动" 验收标准与"全量索引放启动里"互相矛盾
§8 验收 3 说"30 分钟内可完成首次启动并完成一次问答"，但 §6.4 关键流程 1
说 `run.sh` 先做"扫描 Vault → 切分 → Embedding → 写 Chroma"再启动 uvicorn。
对几百篇笔记的用户来说，首次全量 Embedding 用远程 API 就要 5-15 分钟，
MVP 阶段这个数字会卡死验收。

- 建议：把验收 3 拆成两条：
  - 3a) 冷启动（不含全量索引）≤ 3 分钟到能问答
  - 3b) 全量索引时间在 README 给出预期（如 100 篇 ≈ 2 min）
  - 并在 §6.4 增加"先用预建索引 demo 一次问答"作为 onboarding 第一步

### B3. "无明显胡编" 不可验收，且缺一条与"概念解释"对应的硬约束
US4 只覆盖"找不到内容时不胡编"，但 §2.2 场景 3"解释型"（笔记里怎么定义 X）
是 LLM 最容易翻车的——笔记没有明确定义时 LLM 会自己脑补。

- 建议：新增 US6 — "对于笔记中未明确定义的概念，API 必须返回
  '你的笔记里没找到 X 的明确定义，仅有的相关片段是：...'，禁止 LLM
  自由发挥" —— 这条要和 US4 一样写成自动化断言。

## 重要建议（不改也走，但会显著影响后续判断）

### I1. 缺少竞品 / 参考分析
Obsidian 生态已经有 Smart Connections、Copilot for Obsidian 等同类插件，
PRD §1.2 完全没提"和它们差在哪"。原型阶段这是最大的盲点：可能我们花了
力气做出来的 MVP，刚好落在别人已经做得更好的赛道上。

- 建议：§1.2 加一节"差异化定位"，至少明确 1-2 个差异点（本地优先 / 零
  依赖 / 单文件 / 跨平台……）。这一节会反过来影响 §3 功能取舍。

### I2. §10 v0.4 目标是 Obsidian 插件，但 MVP 接口没为它留扩展点
v0.4 要回到 Obsidian 内问答，意味着 chunk_id 迟早要支持 obsidian:// 协议、
vault:// 定位。如果 MVP 把 chunk_id 设计成纯数据库自增 ID，后期改造成本
会很高。

- 建议：§6.3 数据模型里把 `source` 字段从 `相对 data_dir 的路径` 改成
  `vault://<vault-name>/<相对路径>#<heading-or-line>`，F5 的 sources 响应
  用同一套协议。这样 Web 端、Obsidian 插件端、未来 MCP 端共用一套引用。

### I3. 缺"冷启动体验"设计
用户第一次启动时 Vault 还没索引完，能给什么？是空白 + 进度条，还是
"先用几条示例问题试一下"？这是 AI 工具被弃用的高发场景——首跑 30 秒
没回应，用户关掉再也不会回来。

- 建议：§6.4 增加"冷启动"分支，并行做"后台索引 + 前台可先用 5 条示例
  问题演示"。这条不写进 MVP 验收，但要在 §3 留出位置。

### I4. 缺北极星指标，原型"是否值得继续投入" 无法判断
§1.3 说 MVP 目标是"达到可用水平，足以判断后续是否值得继续投入"，但
§8 三个验收项都不能直接回答"值得 / 不值得"。需要至少 1 个**使用行为**
指标作为客观依据。

- 建议：§8 加一条
  - 4) **用户自评信号**：连续 5 天用户主动用该服务提问 ≥ 3 次/天，
    且引用点击率（点开 sources 的次数 / 总回答次数）≥ 30%。

### I5. 性能指标"端到端 10s" 太松，体现不出本地 RAG 的优势
本地 RAG 对比 ChatGPT 的最大卖点是"快到不打断思路"。10s 已经是 ChatGPT
水平，谈不上差异化。

- 建议：§5 把性能拆成两条
  - 首字延迟（流式）：≤ 2s（M1 Pro）
  - 总延迟（非流式）：≤ 10s 保留作为兜底
  - 并配合 §7 增加 `/api/chat/stream`（与开发同事建议一致）

## 文档 / 结构问题

### S1. PRD 与"实现建议"混在一起
§6 直接写死了 LangChain / Chroma / FastAPI / 单 HTML。PRD 应该说"我们要
什么"，"用什么实现"应该交给 ADR（与开发同事意见一致）。建议把 §6 改成
"约束与偏好"：组件数量、依赖管理、单机部署、Python 后端、零构建前端，
剩下的交给 ADR 拍。

### S2. §11 待确认项缺少 owner 和决策时机
5 条都是开放式问题。建议：
- 1、2、5 标 **owner=产品**，**截止=MAQ-5 close 前**（硬阻塞）
- 3、4 标 **owner=产品 + 开发**，**截止=对应 ADR 落地前**（软阻塞）

### S3. §1.1 用户原话与 §6 选型混排，建议在 PRD 头部加一段"读者指南"
让产品同事读 §1-§5（"我们要什么"），让开发读 §6 + ADR（"怎么实现"），
避免同一份文档被两个角色读到一半产生错位预期。
