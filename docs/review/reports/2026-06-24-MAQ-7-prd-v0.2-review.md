## 1. v0.2 新发现的意见

### 阻塞项（建议在 v0.3 修订，不阻塞 in_progress）

> 注：以下 1 条是 v0.2 评审**新提出**的阻塞；与上轮 B1/B2/B3 已闭环不冲突。

#### NB1. §8.2 US6 断言"验证 prompt 包含约束"是弱保证

§8.2 US6 自动化断言写的是 "在 `tests/test_chat.py` 用 mock LLM / mock retrieve
**验证 prompt 模板里包含该约束**"。

- 问题：prompt 里写了"找不到定义就返回 X 文案" 不等于 LLM 实际会按这个文案返回。
  本地小模型（Ollama qwen2.5 / llama3）在此类结构化指令上的服从度并不稳定。
  这个断言通过 不等于 验收通过。
- 建议（v0.3 必改）：把 US6 的"找不到定义"判定从 LLM 行为前移到**检索后处理**层。
  即：在 `retrieve` 返回 hits 后，系统**先**做一次确定性检测
  （`is_defined_in_hits(query, hits) -> bool`），若 `False` 则**根本不发 LLM**，
  直接返回"你的笔记里没找到 X 的明确定义，仅有的相关片段是：..."+ hits。
  这样做的好处：
  1. 断言变成纯函数行为测试（不依赖 mock LLM），真·可验收
  2. 零成本/零延迟（比调 LLM 还快）
  3. 顺手把"未找到相关内容"（US4）和"找到但无定义"（US6）合并成一条
     决策链：`hits 为空 → US4`；`hits 非空但无定义 → US6`；`hits 非空且有定义 → LLM 生成`
- 落地位置：`docs/product/specs/MAQ-5-prd-kb-qa.md` §8.2 US6 断言段 + 在 §3.1 F5
  加一行"生成前预处理：定义存在性检测"。

### 重要建议（建议在 v0.3 修订）

#### NI1. §8.4 北极星指标"上报"方式没说清

§8.4 表第 1 行采集方式写"浏览器本地存储脱敏后**上报**"。上报 = 发 HTTP 请求
到**哪里**？

- 这是本地单机 RAG，不应该有外部网络出口；§5 已明确"默认全程本地"。
  "上报"如果指本机 HTTP POST 到本地后端 → 没问题，但需要点破。
- 建议（v0.3）：把"上报"明确为"POST 到本机 `/api/usage` 端点；本地持久化为
  `data/usage/local-{date}.jsonl`；不上送任何外部服务"。
  并在 §5 隐私一行追加："**所有使用行为埋点仅落本机 `data/usage/`，无任何外发**"。

#### NI2. §7.1 vault:// 协议缺 URL 编码规则

§7.1 例：`vault://my-notes/AI/微服务治理.md#服务治理`。
- 协议对，但没规定：
  1. anchor 是 heading 原文还是 slug 化（`#微服务治理` vs `#服务治理-1`）？
  2. 包含空格 / 特殊字符 / 中文 heading 时如何 URL 编码？
  3. 文件名 / 路径中含空格或中文时如何编码？
- 建议（v0.3）：在 §7.1 追加一段"**编码规则**"：
  - vault-name：RFC 3986 unreserved + 百分号编码
  - relative-path：按 URL path 段编码（`%20` 表空格、中文按 UTF-8 percent-encoding）
  - anchor：heading 原文的**百分号编码**（**不**做 slug 化；前端按原样去匹配
    heading 节点，最简单也最可读）
  - 给一行"建议实现时参照 Python `urllib.parse.quote(..., safe='/')`"。

#### NI3. §3.2 F10 "浏览器本地存储 / SQLite" 自相矛盾

§3.2 F10 写"浏览器本地存储 / SQLite 存最近 N 条对话"。
- 浏览器本地存储和 SQLite 是两套东西：前者是浏览器侧 `localStorage`，
  后者是**服务端**库（Python 端 `sqlite3`）。"A / B"并列没有意义。
- 建议（v0.3）：明确 F10 是 v0.2 阶段（MVP 不做），"Nice to Have"中只挑一个
  默认方案。建议**默认浏览器 `localStorage`**（单机、零运维、和隐私基线一致）；
  留"如需跨设备同步再升 SQLite"作为 §10 v0.3 候选。

### 文档 / 结构建议（建议在 v0.3 修订）

#### NS1. §5 性能行"1k 切片以内"位置尴尬

§5 性能行末括号里出现"1k 切片以内（响应开发 §5）"，阅读时像是脚注残留。
- 建议（v0.3）：把"≤ 1k 切片"从括号里拎出来，作为性能行末尾的明文条件：
  "**以上指标在 Vault 总量 ≤ 1k 切片时成立**；超出后按 §3.1 F3 增量更新策略
  解决，全量重建按 §8.3 3b 预期时间。"

#### NS2. §3 F8 "30 秒内必须有可见反馈" 缺测量点

§3 F8 写"30 秒内必须有可见反馈"。30 秒从哪里开始算？到哪个事件为止？

- 建议（v0.3）：明确测量点：自"前端 `index.html` 首字节"开始；终点 =
  "前端看到 5 条示例问题按钮可点击" 或 "前端看到 `state=building` + 进度"。并
  把这条挂进 §8.2 作为可测断言（`tests/test_cold_start.py` mock 后端 /
  计时前端 → 必须在 30s 内解锁示例按钮）。

#### NS3. §3 F1 "Vault 路径配置" 缺默认路径

§3 F1 只说"用户在配置文件中指定 Vault 根目录"，没说：

1. 配置文件**默认在哪**？`./config.yaml`？`~/.config/rag-demo/config.yaml`？`./.env`？
2. 如果用户**不写配置**怎么办？是用一个示例 Vault 还是报错？

- 建议（v0.3）：在 §3 F1 末尾追加"**默认配置路径**：`./config.yaml`
  （仓库根，相对路径）；如不存在，README 提示用 `config.example.yaml` 起步。
  第一次启动时若 Vault 路径未配置，**走 §3 F8 冷启动 demo 路径**而非报错。"

#### NS4. §3 F3 "ingest --full" 与 §7.2 "/api/ingest" 关系没说清

§3 F3 说"提供 `ingest --full` 全量重建开关"；§7.2 又有 `POST /api/ingest`
触发全量重建。两者是什么关系？

- 建议（v0.3）：在 §6.1.3 之后追加一行"**CLI 与 HTTP API 等价**：
  `rag-demo ingest --full` 等价于 `POST /api/ingest`；两者复用同一份业务函数
  `ingest_directory(data_dir, index_dir, *, full=True)`。Web UI 不直接触发
  全量重建（仅提供状态展示），触发必须经 CLI 或 API 调用方知情同意。"

## 2. 整体评价

v0.2 已经把 v0.1 评审的 3 个阻塞（B1/B2/B3）+ 5 个重要建议（I1-I5）
+ 3 个结构问题（S1-S3）完整消化。文档成熟度从"半成品 PRD"跃升到"可进入
开发 in_progress"。§0 读者指南、§6 约束/ADR 拆分、§11 硬阻塞表三处工程
实践的加入让文档"既能被产品读、也能被开发读、也能被审查员读"。

新发现的 1 个阻塞（NB1）属于 US6 自动化断言**强度**问题，不动摇 MVP
方向；建议在 v0.3 通过"前移到检索后处理"的方式一次解决，比原方案
（依赖 LLM 服从 prompt）更稳、更快、更可测。

3 个重要建议（NI1-NI3）和 4 个结构建议（NS1-NS4）都是一两句话就能补齐的
小补丁，不会影响 ADR 起草节奏。
